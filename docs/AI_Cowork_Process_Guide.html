Process Guide & Retrospective

Building a Productivity System with AI Co-Work

How the Faculty PM System was designed and built iteratively using Claude — what worked, what was missing, and how to do it better next time.

Real retrospective Reusable framework Teaching reference
Section 1

What Was Built & How It Evolved

The system was not designed up-front — it grew iteratively over many sessions. Each session solved a concrete pain point and exposed the next one.

1
Foundation
Core 3-layer Notion architecture
Defined the conceptual model: Project WBS (where you plan) → Master WBS Tasks (where the tool writes) → Work Sessions (time log). Established one-directional sync, idempotent upserts via JSON mapping files, and the principle that status flows upward automatically from Work Sessions.
2
Tool Build
Python/Flask sync tool (notion_wbs_sync.py)
Built a ~3,000-line Flask web app with Setup, Sources, Sync, and Help tabs. The tool discovered WBS databases by title, mapped columns via a config file, and wrote normalized data to Master WBS. Value normalization handled the messiness of different project WBS conventions ("Todo" vs "Not Started" etc).
⚠ What was missing at this stage No context management plan. Sessions grew huge as the tool grew, eventually hitting a 1M-token context limit mid-session. The MANIFEST.md and skill system weren't created until after this problem forced the issue.
3
UX Layer
Quick Start tab — one-click task + session creation
Added a Quick Start tab that creates a task in the Project WBS + Master WBS + Work Sessions in a single click. Later extended to support project-specific fields: Program Management got Category/Phase dropdowns, Professional Services got Service Level and Org/Division (with combobox for new values).
⚠ What was missing at this stage The field variations across projects (some use "Task Type", some "Phase", some "Category") were not documented upfront. Each new project required a separate session to wire up its unique fields, causing repeated discovery work.
4
Hierarchy Fix
Hub page structure standardized
Discovered that hub pages were being placed at the wrong level in Notion's sidebar hierarchy. Research project hub pages should nest under their Research Projects DB row; hub pages must end with a WBS embedded DB + Work Sessions linked table. These rules had to be backfilled across existing projects.
⚠ What was missing at this stage A template or standard was never defined before creating the first hub page. The structure was discovered through trial and error, then retroactively applied. Defining hub page templates before creating any projects would have saved multiple correction sessions.
5
Context Crisis
Context limit hit → MANIFEST + skills created
A long session adding Professional Services fields hit the 1M-token limit mid-task and had to be abandoned. This forced the creation of: (1) _MANIFEST.md — a tiered file index telling Claude which files to read and when; (2) skill files that encode pre-computed knowledge (DB IDs, field maps, line numbers) so each session doesn't re-derive everything from scratch.
⚠ What was missing at this stage Context management should be designed on Day 1 of any AI co-work project, not after a crisis. A MANIFEST and a "what does Claude need to know per session" plan would have prevented this entirely.
6
API Discovery
Auto Status formula unreadable via Notion API
The focus task list feature revealed that the Notion MCP returns formula field values as opaque formulaResult://... URLs, not the computed string. Tasks that were "Completed" appeared as still pending. The workaround: fetch each task's linked Work Sessions and inspect their Status fields directly — accurate but adds extra API calls.
⚠ What was missing at this stage No API constraint audit was done before designing the Auto Status formula as the primary completion signal. If Notion API limitations had been surveyed first, the formula would have been designed differently (or a workaround pre-planned).
7
Intelligence Layer
Skills, focus task list, workload dashboard
Built reusable skills encoding domain knowledge: notion-workload-query (correct query patterns to avoid fetching all 241 sessions), add-wbs-config-entry (schema + common mistakes pre-encoded), focus-task-list (live prioritized task view on Notion Home), and sync-tool-edit (section map for the 3,000-line Python file). These make future sessions dramatically more efficient.
Section 2

Root Causes of Rework

Every correction session was traceable to one of five root causes. Understanding these lets you prevent them in future projects.

🏗️
No upfront data model
Caused: hierarchy mistakes, retroactive session creation, backfilling rules
The three-layer model (WBS → Master → Sessions) was sound, but the relationships between entities weren't fully drawn before building. Questions like "what does a hub page contain?", "what does a Projects DB entry look like?", and "how do research projects differ from teaching projects?" were answered during construction, not before. Each answer required backfilling existing records.
📋
No template library defined at the start
Caused: inconsistent hub pages, repeated structural corrections
Hub page structure, WBS database schema, and Work Sessions section placement should have been a locked template before the first project was created. Instead, the standard was reverse-engineered from the best-looking existing page ("model page: Beyond the LXD Label hub") and then applied backward.
🔌
No API constraint audit
Caused: Auto Status formula workaround discovered late
The Notion API's formula field limitation (returns opaque URL, not computed value) was a known category of risk but wasn't tested before building the focus task list around it. A 30-minute API constraint audit — fetch a test formula field and inspect the raw response — would have revealed this before it became a bug.
🗂️
No context management plan
Caused: 1M token limit hit, abandoned session, MANIFEST created reactively
AI co-work sessions accumulate context with every tool call. A 3,000-line file + config files + Notion API responses = context explosion. The MANIFEST.md tiering system and skill files were the right solution — but they should be created on Day 1, not after a crisis forces the issue.
🔄
Scope creep within single sessions
Caused: abandoned sessions, partial implementations
The session that hit the context limit was trying to add multiple fields (Level + Org/Division) in one go, on top of an already-large conversation. Each session should have a single, concrete deliverable. "Add Level field to Quick Start" is a session. "Add Level and Org/Division and update the config and test it" is three sessions.
Section 3

The Improved AI Co-Work Process

A seven-phase framework for building a complex productivity system with AI. Each phase has a concrete deliverable that the AI can help produce.

  • 1
    Phase: Problem & Goal Scoping

    Before touching any tools, write a plain-language brief: What problem are you solving? Who uses this? What does "done" look like? What are the constraints (budget, tools, time)?
    AI role: Interview you about the problem space. Push back on vague goals. Produce a one-page brief you both agree on before any code.

  • 2
    Phase: Data Model & Entity Design

    Draw the entities, their fields, and their relationships. Decide on IDs, statuses, and what triggers what. This is the most important phase — every correction in the Faculty PM project traced back to skipping this.
    Deliverable: Entity diagram + field table. Every relationship is explicit. Every status transition is documented.

  • 3
    Phase: API & Platform Constraint Audit

    Before designing anything that reads from or writes to an external API, audit what the API can and cannot do. Fetch a sample response for every field type you plan to use. Check: formula fields, relation fields, rollup fields, file fields.
    For Notion specifically: Test formula field responses. Test relation filter behavior. Test what happens when you clear a filter via API. Document the gaps.

  • 4
    Phase: Template & Standard Library

    Before creating any instances, create the template. For a Notion-based system: define the hub page template, the WBS database schema template, the field naming conventions. Lock these before creating any projects.
    Why: Retroactively applying a standard across 5+ existing projects is 5× harder than setting it once at the start.

  • 5
    Phase: Context Management Architecture

    On Day 1, decide how you'll manage AI session context for the life of the project. Create:
    (a) _MANIFEST.md — file index with tiered read policies
    (b) Skill files for every domain the AI will repeatedly work in
    (c) Session size discipline: one deliverable per session, fresh start for each task
    Rule of thumb: If a session makes more than 8-10 tool calls, consider splitting it.

  • 6
    Phase: Incremental Build (Vertical Slices)

    Build the system in vertical slices — each slice is end-to-end functional, even if narrow. Slice 1 might be "sync one project's tasks to Master WBS." That's it. Test it. Then add Slice 2 (Work Sessions). Then Slice 3 (Quick Start).
    Never: Build the full schema + all projects + the UI + automation all in one session. Each session = one slice.

  • 7
    Phase: Knowledge Crystallization

    After each significant build session, have the AI write a retrospective: what invariants were discovered, what rules were established, what edge cases were handled. This feeds back into skills and memory so future sessions don't re-derive it.
    Deliverable: Updated skill files, memory entries, and a design doc (like this one).

Section 4

Principles for Effective AI Co-Work

Distilled from the Faculty PM System build. Applicable to any complex project with an AI collaborator.

💡
Treat the AI like a brilliant new hire, not a search engine. Give context, not just commands. The more you explain why something needs to be a certain way, the better the AI's judgment on edge cases you didn't anticipate.
⚠️
Context is the most finite resource. Every tool call, every file read, every API response costs context tokens. Design your project so the AI can complete a useful task in 5–8 tool calls. If it regularly takes 15+, your session scope is too broad.
📐
Never spec in the building phase. If you're discovering requirements while the AI is coding, you're wasting time. Finish the spec (even a rough one) before the first line of code. The AI is very good at helping you write the spec — use it for that first.
01

One Session, One Deliverable

Define the single thing this session will produce before starting. Not "improve the system" — "add a Level dropdown to Professional Services Quick Start." When it's done, stop.

Bad: "Add Level, Org/Division, update config, and test"   Good: "Add Level dropdown to Quick Start (one session), then Org/Division (next session)"
02

Write Invariants, Not Just Requirements

For every rule the system must follow, write it as an invariant: "Status is ONLY written on CREATE, never on UPDATE." This framing forces you to think about edge cases before they become bugs.

Example: "Deleted source tasks are NOT removed from Master WBS" — this isn't obvious from requirements, but it's a critical invariant for data integrity.
03

Test the API Before Designing Around It

Any time your design depends on a specific API behavior, verify it empirically before committing. Do a quick test call and inspect the raw response. Don't trust documentation or assumptions.

Cost of not doing this: The Notion formula field limitation wasn't caught until the focus task list was already built around it.
04

Skill-ify Repeated Work

Any task you expect to repeat more than twice should become a skill. "Create a new project" became the notion-new-project skill. This moves the knowledge out of your session context and into a reusable file.

Rule of thumb: If you find yourself re-reading the same file at the start of multiple sessions, that file's content should be a skill.
05

Ship a Vertical Slice First

Start with the simplest version that's fully functional end-to-end. One project, one WBS, one sync, one session. Prove the architecture works before adding complexity.

For the Faculty PM System: "Sync EDG 6648 WBS to Master WBS Tasks" should have been v1. The multi-project config came after proof of concept.
06

Document Decisions, Not Just Outcomes

Every non-obvious design decision should be documented with its rationale. Future sessions — and future you — need to know why, not just what. Skills and memory files are the right place.

Example: "Status written only on CREATE — protects Auto Status formula from being overwritten by sync" explains both the rule and why breaking it causes a bug.
Section 5

"Before You Start" Checklist for AI-Assisted System Projects

Use this checklist at the start of any new productivity/automation project you're building with an AI collaborator. These are the questions this project had to answer reactively — answer them proactively.

🏗 Architecture & Data Model

  • What are the core entities? What fields does each have?
  • What are the relationships between entities? (one-to-many, many-to-many?)
  • What are the unique identifiers for each entity?
  • What triggers what? (e.g., "completing a session triggers a status update")
  • What is the canonical source of truth for each field?
  • What are the status lifecycle states and valid transitions?

🔌 Platform & API

  • Which field types am I using? Have I fetched a raw API response for each?
  • What actions are NOT supported via API? (formula reads, relation filters, file uploads?)
  • What are the rate limits and pagination behaviors?
  • What API responses does "success" look like? What does "silent failure" look like?

📋 Templates & Standards

  • What does a standard "project" look like? (hub page, database, linked tables)
  • What naming conventions will I use for databases, fields, and files?
  • What is the required structure of every instance? (e.g., every hub page ends with WBS + Work Sessions)
  • Can I create one complete "model" instance before creating any others?

🗂 Context & Session Management

  • Which files will the AI need to read repeatedly? → Put them in Tier 1 of MANIFEST.
  • Which knowledge will be needed in every session? → Make it a skill.
  • What is the largest single file in my project? → The AI should never read it in full.
  • What is the session size discipline? (one deliverable per session)

📐 Invariants & Rules

  • What can NEVER happen? (e.g., never edit Master WBS directly)
  • What must ALWAYS happen? (e.g., every project must have a Projects DB entry)
  • What is the idempotency strategy? (how do repeated runs stay safe?)
  • What is the data retention policy? (deleted records — remove or archive?)
