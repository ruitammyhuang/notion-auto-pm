Focal — Faculty Project & Workload Hub

Automated project management for a multi-faceted faculty role — built on Notion + Python/Flask

System Overview

Problem

A faculty role spans simultaneous, structurally different workstreams — courses, research projects, service roles, admin — each with its own planning context. Generic PM tools may not accommodate the diverse structure faculty work actually has. Managing projects in separate Notion WBS databases solves that, but creates a new problem: no unified view, no cross-project status, no way to see what actually needs attention this week.

What Focal does

It bridges per-project Notion WBS databases into a single consolidated system — without restructuring how each project is planned. Tasks sync automatically into a Master task list. Work sessions tie actual time logged to specific tasks and projects. Status propagates up automatically once a session is marked complete.

Core features

🔄 Sync
Pulls tasks from all project WBS databases into one Master list; run on demand or on a schedule.
⚡ Quick Start
Single entry point to log a work session or add a task to any project, without navigating to each WBS.
📌 Focus view
Shows what's overdue and due this week, ranked by urgency, across all work areas.
📊 Workload dashboard
Actual hours logged by project and work type over a selected time window.
Three-Layer Architecture
1
📋 Project WBS Databases (Planning Portal)
One dedicated Notion database per project. Author and edit tasks here — add rows, set due dates, adjust priorities, track phases. Currently 8 active WBS databases covering all work areas. Lives on each project hub page.
User edits here · Source of truth
↓ Python sync pushes changes
2
📋 Master WBS Tasks (Implementation Tracking)
Single aggregated Notion database consolidating tasks from all project WBSes into one canonical list. The Python sync tool writes here — never edit directly. Powers cross-project views, focus lists, and status rollups.
Auto-written by sync tool · Never edit directly
↓ Sync creates sessions · Status propagates up
3
⏱️ Work Sessions (Time-Logging Layer)
Time log entries linked to both a task (Master WBS) and a project (Projects DB). Created via Quick Start. The Status field on sessions drives the Auto Status formula in Master WBS, which rolls up to Project WBS via relation.

Sync Tool

Focal (main.py) is a local Flask web app at localhost:8765. One-directional bridge: reads from Project WBS databases, writes to Master WBS Tasks and Work Sessions. State is persisted in JSON mapping files for idempotent upserts.

Process Flow
1

Edit tasks in Project WBS

Add/update rows directly in Notion. Set Task name, Due Date, Priority, Notes, Phase/Category. This is the only manual step.

2

Run sync tool → Sync tab → Sync Now

Reads all configured WBS databases via Notion API. Maps columns using field_map in config. Normalizes status/priority values.

3

Upsert into Master WBS Tasks

Checks focal_mappings.json for existing mapping. New → CREATE + write backlink. Existing → UPDATE changed fields only. Status set ONLY on CREATE.

4

Create Work Session (Quick Start)

Via Quick Start tab: creates session linked to Master WBS task + Projects DB entry. Mapping persisted in focal_sessions_mappings.json.

5

Status propagates automatically

Mark Work Session Status = “Completed” in Notion → Master WBS Auto Status formula updates → Project WBS Auto Status rollup reflects completion.

Key Sync Behaviors
Idempotent Upserts

Mapping files track every synced task/session. Re-running sync updates existing records — never creates duplicates.

One-Directional Only

Project WBS → Master WBS. Changes to Master WBS directly are NOT reflected back. Always edit in the source WBS.

Status Protection

Status is written ONLY on CREATE in Master WBS. Never overwritten on UPDATE — protects the Auto Status formula from being clobbered.

Value Normalization

Status and Priority strings normalized on sync: “Todo / Not Started” → Not Started · “Done / Finished” → Completed · “Urgent / Critical” → Urgent.

Project Placement Rules
  • 📁Always create a 📁 Projects DB entry for every new project — this is what Work Sessions “Project” relation links to. Tagged by category.
  • 🔬Research projects get two records: a 📁 Projects DB entry (for time logging) AND a 🎍 Research Projects DB entry (for research tracking). Hub page nests under the Research Projects DB row.
  • 📍Hub page placement is separate from DB entry. Place hub pages nested under their section’s DB row in the sidebar hierarchy.

Known API Limitation
⚠️
Auto Status formula not readable via API. The Notion MCP returns formula values as opaque formulaResult://… URLs. To check task completion programmatically: fetch the task’s linked Work Sessions, then check each session’s Status field. A task is done when all non-deleted sessions are “Completed.”
Config Schema — focal_config.json

One entry per source WBS database. The sources key is a dict keyed by WBS Notion DB ID (UUID with hyphens).

"sources": {
  "<wbs-db-uuid>": {
    "db_title": "WBS — Project Name", // display name
    "project_id": "<projects-db-entry-uuid>", // 📁 Projects DB entry ID
    "backlink_field": "Master WBS", // relation column in Project WBS
    "auto_calc_planned_start": true,
    "field_map": {
      "task_name": "Task", // REQUIRED
      "priority": "Priority",
      "planned_end": "Due Date",
      "notes": "Notes",
      "category": "Phase" // "Category", "Phase", etc.
    }
  }
}
Design Rules & Invariants
  • 🔒Never edit Master WBS Tasks directly. All changes must originate in a Project WBS and flow through the sync tool.
  • 🔁Status written only on CREATE. The sync tool never overwrites Status on UPDATE — intentional to protect the Auto Status formula chain.
  • 🗂️Every project needs a 📁 Projects DB entry. This is the anchor for Work Sessions. Without it, sessions cannot be assigned to a project.
  • 🔗backlink_field must always be “Master WBS”. This relation column links Project WBS rows back to Master WBS Tasks entries. Auto-written by the tool on first sync.
  • 🧩Config keys must be UUID-with-hyphens format. e.g. 001e7ca9-a7b8-4180-…, not the compact form.
  • 🚫Deleted source tasks are NOT removed from Master WBS. Preserves historical log integrity.
